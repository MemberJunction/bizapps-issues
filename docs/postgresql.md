# PostgreSQL Support

BizApps Issues installs into a MemberJunction database on **either SQL Server or PostgreSQL**
(including AWS Aurora / RDS, Cloud SQL, and Azure Database for PostgreSQL), mirroring how
MemberJunction itself supports both platforms.

The canonical migrations are authored once in SQL Server T‑SQL under `migrations/`. A converted
PostgreSQL set lives under `migrations-pg/`. A single environment variable selects which set runs.

---

## TL;DR — install on PostgreSQL

```bash
# 1. point at a PostgreSQL MJ instance
export DB_PLATFORM=postgresql
# DB_HOST/DB_PORT(5432)/DB_DATABASE/DB_USERNAME/DB_PASSWORD + CODEGEN_DB_* as usual

# 2. three-stage install (same shape as MemberJunction core on PG)
npm run mj:migrate            # DDL — runs migrations-pg/ automatically (folder auto-swap)
npm run mj:codegen            # regenerates CRUD functions, views, triggers as native PG
npx mj sync push --dir ./metadata   # seeds reference data (Issue Statuses + Types)

# 3. build + start
npm run build
npm run start
```

On SQL Server the flow is identical without `DB_PLATFORM` (defaults to `sqlserver`) and runs
`migrations/`.

---

## How platform selection works (inherited from the MJ CLI — no custom code)

The mechanism is entirely provided by `@memberjunction/cli` (5.40.2). BizApps Issues does **not**
reimplement any of it:

- **`DB_PLATFORM`** (`sqlserver` | `postgresql`) is read by the CLI. There is no new config key to set
  — do **not** hardcode `dbPlatform` in `mj.config.cjs`, because the file value would override the
  env var and defeat per-environment switching.
- **`mj migrate`** swaps the migrations folder `migrations` → `migrations-pg` automatically when
  `DB_PLATFORM=postgresql`, and dispatches to the PostgreSQL Skyway provider. The `.pg.sql` suffix is
  cosmetic — dialect selection is purely by folder.
- The **runtime** (MJAPI) selects `@memberjunction/postgresql-dataprovider` from the same
  `DB_PLATFORM` env var. That provider ships transitively with `@memberjunction/server`, so no extra
  runtime dependency is required.

`npm run mj:migrate` resolves the app's `${flyway:defaultSchema}` placeholder to the app schema
(`__mj_BizAppsIssues`) and `${mjSchema}` → core `__mj`, exactly as the OpenApp installer does.

## Keeping `migrations-pg/` in sync — the two‑command process

When you add or change a SQL Server migration under `migrations/`, regenerate its PostgreSQL twin:

```bash
npm run mj:migrate:convert      # mj migrate convert + scripts/pg-finalize.mjs (one combined script)
```

`mj:migrate:convert` runs two steps in sequence:

1. **`mj migrate convert`** (MemberJunction's rule-based SQL converter) does the bulk SS→PG
   translation and writes `migrations-pg/*.pg.sql`. It **skips** any migration that already has a
   `.pg.sql` twin, so previously-shipped (and finalized) migrations are never rewritten. It is
   invoked with explicit `--source-dir ./migrations --output-dir ./migrations-pg --schema
   __mj_BizAppsIssues` because the CLI defaults (`./migrations/v5`) do not match this repo's flat
   layout.
2. **`scripts/pg-finalize.mjs`** applies a small set of deterministic, **idempotent** patches for the
   handful of patterns the converter does not fully handle. Re-running it on already-finalized files
   is a no-op. This is the "things we fix after the converter" layer — it is *code*, not hand-editing,
   so the process stays repeatable for every future migration.

If a brand-new migration uses a construct none of the existing rules cover, either add one rule to
`scripts/pg-finalize.mjs` (preferred — it then applies forever) or hand-author a `*.pg-only.sql`
supplement (MemberJunction's sanctioned manual-override mechanism for migrations the converter
cannot produce).

### What `pg-finalize.mjs` fixes (and why)

| # | Patch | Converter gap it compensates for |
|---|-------|----------------------------------|
| 1 | Quote FK-join alias references | Converter quotes the `AS "alias"` definition but leaves references unquoted → PG case-folds them → "missing FROM-clause entry". |
| 2 | Neutralize skipped triggers / unrecognized batches | Converter comments only the `CREATE TRIGGER` line (or marks a batch `UNKNOWN`) but leaves invalid T‑SQL bodies live. These are regenerated by `mj codegen` on PG. |
| 3 | Cast core boolean INSERTs | Converter emits native `TRUE/FALSE` for bizapps tables it sized, but `0/1` for core `__mj` tables it can't see. (Map in `scripts/pg-core-boolean-columns.json`.) |
| 4 | Cast boolean comparisons | `"boolCol" = 0/1` → `= FALSE/TRUE` (PG has no implicit boolean=integer operator). Scoped to known bool columns (`IsActive`, `IsDefault`, `IsTerminal`, `IsResolved` ∪ the MJ-core map). |
| 5 | Quote core object refs | Unquoted mixed-case `__mj.vwX` → folds to a non-existent lowercased relation. |
| 6 | Drop-before-create views | `CREATE OR REPLACE VIEW` cannot rename/reorder columns; a regeneration migration reshapes a view → 42P16. We DROP first so the create is fresh (`mj codegen` is the authoritative source for views). |
| 7 | Cast `_Clear` flag args | CodeGen "tolerant" CRUD sprocs take boolean `<field>_Clear` flags; the converter passes integer `:= 1/0`, which won't match the boolean parameter. |
| 8 | Defer bizapps sproc seed blocks | Metadata_Sync seeds the status/type tables by calling `__mj_BizAppsIssues."spCreate*"`, which don't exist until `mj codegen` runs. The same seed data ships in `metadata/` and is loaded by `mj sync push` after codegen. |
| 9 | Normalize bizapps schema unquoted | Converter leaves the schema name unquoted (PG folds to lowercase `__mj_bizappsissues`); `mj codegen` and the runtime also emit the schema unquoted, so we keep it consistently unquoted to avoid "relation does not exist". |

See `scripts/pg-finalize.mjs` for the exact, commented rule set. The patches are adapted from the
bizapps-common pipeline; they are generic — only the app schema name (`__mj_BizAppsIssues`) and the
bizapps boolean-column list differ.

## Why three stages on PostgreSQL

MemberJunction's PostgreSQL model is **DDL via migrations → CRUD functions/views/triggers via
`mj codegen` → reference data via `mj sync push`**. The CodeGen-generated stored procedures, base
views, and update triggers embedded in the SQL Server migrations are *regenerated natively* on
PostgreSQL by `mj codegen` (as PL/pgSQL functions and PG triggers), not translated. That is why the
finalize step neutralizes those sections rather than attempting a faithful T‑SQL→PL/pgSQL port, and
why the type/status-table seed data is applied by `mj sync push` after codegen has created the CRUD
functions.

## Version note — why MJ 5.40.2

BizApps Issues pins `@memberjunction/*` at **5.40.2**, the same floor bizapps-common verified for
PostgreSQL. App-schema `mj sync push` (seeding the status/type tables via app-schema `spCreate*`
functions) was broken on MJ ≤ 5.37.0 — the PostgreSQL data provider hardcoded the core `__mj` schema
for every CRUD call, so an app-schema seed resolved to a non-existent `__mj.spCreateIssueStatus` and
silently seeded nothing. Fixed in 5.39.0+ (`PostgreSQLDataProvider` now qualifies the call with the
entity's own schema).
